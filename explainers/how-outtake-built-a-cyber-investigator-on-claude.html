The problem

Agentic offense needs agentic defense

When targeting a company, attackers move through the same predictable sequence — now accelerated by AI at every step.

Instead of just taking down a cloned login page, the Recon Agent gathers and classifies evidence from the impersonation event, follows leads to connected infrastructure (like a fake “Customer Support” Telegram account), and maps the adversarial network in a graph. Its final report explains the investigation, profiles the threat actor, and reconstructs a timeline of what the attacker did. It can read, write, and run code — and even interact with malicious login pages to see where stolen credentials actually go.

The build

Building a complex long-running agent, in four stages

Each stage was about understanding what a good investigation looked like — then progressively handing that judgment to the agent.

1
Define “good”

Become the expert first

Engineers ran real cyber investigations themselves and pulled domain expertise from customers and design partners — identifying what evidence matters and what separates an actionable conclusion from a guess. That standard became the fixed reference point for every later stage.

2
Validate

Prototype in Claude Code

The agent needed to write, run code, build tools on the fly, and interact with malicious domains. Claude Code was a strong initial harness to validate those assumptions — and where they forged their core principle: constrain tightly at the orchestration level, but leave the agent free to improvise when judgment is required.

3
Productionize

Graduate to the Agent SDK

Carrying over skills and patterns from Claude Code, the team gained tighter control over the Recon Agent’s memory, context, and file system — without reinventing the agent loop or session handling, and without dropping velocity.

4
Iterate

Build a tight loop driven by evals

Evals were integrated from the very beginning, running many scenarios at once so the team could make sweeping changes — model upgrades, full memory-system refactors — safely. When the agent reports it could have done better with a missing tool, a separate coding agent writes it, tests it, and only then does a human check the result.

Learnings

Four lessons from building a long-running agent

Elaborate scripted workflows are being replaced by a harness: a supportive environment of memory, tools, skills, and guardrails.

📁

A filesystem and bash is all you need

The filesystem enables memory that survives compaction. Giving the agent open-ended tools — read, write, run code — lets it improvise around obstacles. When a tool failed on a network hiccup, the agent simply found a workaround and continued to a successful outcome.

💬

Prompts are suggestions

“As this agent runs longer, every single word in that prompt will probably be ignored eventually.” The fix: identify what the agent should always do, and pull it out of the prompt into the harness — freeing context and attention for where the agent thrives.

⚡

Evals are for speed, not just reliability

Reading a 30-minute agent transcript by hand doesn’t scale. An eval is a structured, graded, automatable version of that reflection — put an agent in the judge’s seat to score the run. Even imperfect evals from day one make you build the agent faster.

🛡

Protect your agents

Prompt injection is a real threat, so Outtake builds a “blastbox” that assumes the agent might get hijacked and contains the damage. They now score trust at the exact moment the agent reaches out to the internet — the armor agents need on an adversarial web.

Field guide

Best practices from the Outtake team

Ask yourselfWhat to do
Do you know what “good” looks like?Be the agent first. Run the real task yourself and pull domain expertise from customers and partners, so you have a fixed standard to hold every iteration against.
Is each piece of complexity earned?Find the simplest working version and automate piece by piece. Add complexity only when results justify it — the same discipline as traditional software.
Is your harness matched to the workload?Validate assumptions fast in Claude Code, then graduate to the Agent SDK when you need lower-level control over memory, context, and sessions. Don’t rebuild the agent loop yourself.
Where should the agent be constrained?Hardcode guardrails at the orchestration layer, but don’t let those constraints reach into low-level judgment calls. The improvisation space is where the best results come from.
